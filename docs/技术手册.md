# 会务管理系统 · 技术手册

面向接手/部署本系统的工程师。涵盖技术栈、环境搭建、数据库初始化、目录结构、关键约定与生产部署。

---

## 1. 项目简介

单会议的会务管理系统:面向参会者的**前台站点** + 面向组织者的**管理后台**,全部内容通过后台维护。

- 前台:欢迎致辞、会议通知、注册报名、论文投稿、简明/详细日程、讲者查询、会场交通、联系方式、酒店预订、图片直播、直播(外链)、个人中心。
- 后台:仪表盘、站点设置、通知/内容页/讲者/日程/酒店管理、报名/论文审核、预订管理、图片直播管理、用户角色管理、报名/投稿 CSV 导出。

## 2. 技术栈

| 层 | 选型 |
|---|---|
| 框架 | Next.js 16(App Router,`src/` 目录,路由分区 `(public)` / `(admin)`) |
| 语言 | TypeScript |
| 数据库 | PostgreSQL(经 Prisma 7 **driver-adapter** `@prisma/adapter-pg` 接入) |
| ORM | Prisma 7 |
| 鉴权 | Auth.js v5(next-auth@beta),Credentials provider,JWT 会话,角色 `USER`/`ADMIN` |
| 密码 | bcryptjs | 
| 校验 | zod |
| 样式 | Tailwind CSS |
| 测试 | Vitest |
| 上传 | 本地磁盘 `public/uploads/`(PDF、图片),路径存库 |

> ⚠️ 这是 **Next.js 16**,若干 API 与旧版不同(见 §9 关键约定)。改动前请参考 `node_modules/next/dist/docs/`。

## 3. 环境要求

- Node.js **20+**
- PostgreSQL **14+**(开发用 18 验证过),需 `psql` 客户端
- npm(仓库自带 `package-lock.json`)

## 4. 快速开始

```bash
# 1) 安装依赖
npm install

# 2) 准备环境变量
cp .env.example .env
#    编辑 .env,把 DATABASE_URL 改成你的 PostgreSQL 连接串,AUTH_SECRET 换成随机串

# 3) 初始化数据库(建库 + 建表 + 种子)
bash scripts/init-db.sh

# 4) 启动开发服务
npm run dev
#    打开 http://localhost:3000
#    后台 http://localhost:3000/admin  (先登录管理员)
```

默认管理员:**admin@conf.local / admin123**(种子创建,首次登录后请改密/改角色策略)。

## 5. 环境变量(`.env`)

| 变量 | 说明 | 示例 |
|---|---|---|
| `DATABASE_URL` | PostgreSQL 连接串 | `postgresql://用户:密码@主机:5432/库名` |
| `AUTH_SECRET` | Auth.js 会话签名密钥,**生产必须**改随机串 | `openssl rand -base64 32` 生成 |

`.env` **不入库**(已在 `.gitignore`)。模板见 `.env.example`。

## 6. 数据库初始化

三种方式任选:

**A. 一键脚本(推荐)**
```bash
bash scripts/init-db.sh
```
自动:创建库(若不存在)→ `prisma db push` 建表 → 写入种子。脚本从 `.env` 解析连接信息,幂等可重复运行。

**B. 手动(Prisma)**
```bash
# 库需先存在(psql -c "CREATE DATABASE huiwu_dev;")
npm run db:push     # 按 schema 建表/同步
npm run db:seed     # 写入种子数据
```

**C. 纯 SQL(交给 DBA / 无 Node 环境建表)**
```bash
psql -h <host> -U <user> -d <db> -f scripts/schema.sql
```
`scripts/schema.sql` 是**由 Prisma schema 自动生成**的建表 DDL(14 张表 + 约束/索引)。注意:此方式只建表结构,种子数据仍需 `npm run db:seed` 或手工插入。

> 修改数据模型后:改 `prisma/schema.prisma` → `npm run db:push`(开发)。若要重新生成 SQL:
> `npx prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script > scripts/schema.sql`

### 数据模型(14 张表)
`User`、`SiteConfig`(单行 id=1)、`Page`、`Notice`、`RegistrationType`、`Registration`、`Submission`、`Speaker`、`Session`、`SessionSpeaker`(复合主键)、`Hotel`、`HotelBooking`、`Album`、`Photo`。外键策略:用户/类型/酒店相关的引用为 `Restrict`(有关联时禁止删除),`SessionSpeaker`/`Photo` 为 `Cascade`。

## 9. 关键约定 / 架构要点

- **读写分离**:前台/后台**页面**是 Server Component,直接 `import { prisma } from "@/lib/prisma"` 读库;**写操作**一律走 `app/api/**/route.ts`,经 zod 校验 + 鉴权后落库,统一返回 `{ ok, error }`。
- **Next.js 16 异步 params**:动态路由的 `params`、页面的 `searchParams`、`RouteContext<'…'>.params` 都是 `Promise`,必须 `await`。
- **后台鉴权两道**:
  1. `middleware.ts` 的 matcher 是 `["/admin/:path*"]`,拦截后台**页面**,非 ADMIN 跳 `/login`。
  2. **`/api/admin/*` 不在该 matcher 内**,因此**每个后台接口 handler 第一条语句必须**自带 `isAdmin(session?.user?.role)` 守卫返回 403。新增后台接口时务必遵守。
- **原生表单 + 303**:后台增删改/审核用原生 HTML `<form method="post">` 提交到接口,处理后 `303` 重定向回列表(无需客户端 JS)。前台创建类表单是 client 组件用 `fetch`。
- **上传**:仅 `application/pdf`(≤10MB)/ `image/jpeg|png|webp`(≤5MB);文件名服务端生成(UUID),存 `public/uploads/`(gitignore)。
- **直播**:采用**外链跳转**(读 `SiteConfig.liveUrl`,`<a target="_blank" rel="noopener noreferrer">`),不做 iframe 内嵌。
- **富文本**:通知/内容页/致辞/讲者简介等为管理员可信 HTML,统一经 `RichText` 组件渲染(集中的 XSS 边界)。
- **CSV 导出**:`toCsv` 对以 `= + - @` 开头的字段前置单引号,防表格软件公式注入。
- **报名唯一**:每个用户仅一条报名,DB 级 `@unique` + 应用层双保险。

## 10. 测试

- 测试用 Vitest,**领域测试直连数据库**,通过 `tests/setup.ts` 加载 `.env.test`。
- 需要一个**独立测试库**(切勿指向开发/生产库,测试会增删数据)。步骤:
  ```bash
  # 建测试库
  psql -h <host> -U <user> -d postgres -c "CREATE DATABASE huiwu_test;"
  # .env.test 内容:
  #   DATABASE_URL="postgresql://用户:密码@主机:5432/huiwu_test"
  #   AUTH_SECRET="test-secret"
  # 建表
  DATABASE_URL="postgresql://用户:密码@主机:5432/huiwu_test" npx prisma db push
  # 跑测试
  npm test
  ```
- 纯函数测试(zod schema、`toCsv`、`validatePdf`、日程分组等)不需要数据库。

## 11. 生产部署要点

1. **构建启动**:`npm run build && npm run start`(或用进程管理器 pm2/systemd 守护)。
2. **环境变量**:生产 `DATABASE_URL` 指向生产库;`AUTH_SECRET` 用强随机串;设 `NODE_ENV=production`。
3. **数据库初始化**:生产库执行 `npm run db:push`(或 `scripts/schema.sql`)+ 视需要 `db:seed`(至少要有 SiteConfig 与一个管理员;生产可只跑管理员+站点配置部分,或手工建管理员)。
4. **连接池**:若部署到 **Serverless / 多实例**,务必用连接池(PgBouncer,或 Neon/Supabase 提供的 pooled 连接串),避免连接耗尽。
5. **上传持久化**:`public/uploads/` 是**本地磁盘**。单机部署可直接用;**Serverless/多实例/容器无持久卷**时本地文件不共享也不持久,需改为对象存储(S3/OSS),替换 `src/lib/upload.ts` 的落盘实现并把公网 URL 存库。
6. **HTTPS / 反代**:前置 Nginx/Caddy 终止 TLS,转发到 Node(默认 3000)。
7. **首个管理员**:由种子创建 `admin@conf.local / admin123`,上线后**立即改密**(或用后台"用户管理"提升某真实账号为 ADMIN 后停用默认号)。

## 12. 常见问题

- **`npm test` 报数据库连接错误**:未配测试库或 `.env.test` 未指向 PG。见 §10。
- **`prisma db push` 认证失败**:检查 `.env` 的 `DATABASE_URL` 用户名/密码、`pg_hba.conf` 是否允许该主机/方式。
- **构建报字体网络错误**:字体已本地化为 `public/fonts/`(离线可构建),无需联网。
- **后台接口返回 403**:确认登录账号 `role='ADMIN'`;新增接口是否加了 `isAdmin` 守卫(见 §9)。
- **上传的文件访问 404**:确认 `public/uploads/` 存在且进程有写权限;生产多实例见 §11.5。
