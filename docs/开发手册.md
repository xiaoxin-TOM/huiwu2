# 会务管理系统 · 开发手册

本手册面向接手、扩展或维护本系统的开发者，重点总结当前系统的**功能架构、模块划分、数据模型、业务操作流程与技术实现现状**。环境搭建、部署细节、数据库初始化等请参阅《技术手册.md》。

---

## 1. 项目定位与总体架构

### 1.1 系统定位

本系统是一个基于 **Next.js App Router** 构建的**单实例多会议会务管理平台**，服务于两类角色：

- **前台用户**：会议参会者、论文投稿人、受邀嘉宾。
- **后台管理员**：会议组织者、会务工作人员、系统管理员。

系统支持在同一部署实例中管理多个会议，每个会议拥有独立的站点配置、内容、报名、日程、酒店、嘉宾、相册等数据；通过"默认会议"机制对外提供无会议 ID 的访问入口。

### 1.2 技术架构

```
┌─────────────────────────────────────────────────────────────────┐
│                          浏览器端                                │
│   前台站点 (mobile-first)          管理后台                      │
└───────────────┬───────────────────────────────┬─────────────────┘
                │                               │
     ┌──────────▼──────────┐      ┌─────────────▼──────────────┐
     │  src/app/(public)   │      │   src/app/(admin)          │
     │  Server Component   │      │   Server Component          │
     │  直连 Prisma 读取    │      │   直连 Prisma 读取          │
     └──────────┬──────────┘      └──────────────┬─────────────┘
                │                                │
                └────────────────┬───────────────┘
                                 │
     ┌───────────────────────────▼────────────────────────────┐
     │              src/app/api/**/route.ts                   │
     │     写操作接口：zod 校验 + auth 鉴权 + Prisma 写入       │
     └───────────────────────────┬────────────────────────────┘
                                 │
     ┌───────────────────────────▼────────────────────────────┐
     │                    Prisma ORM                          │
     │               PostgreSQL (pg adapter)                  │
     └────────────────────────────────────────────────────────┘
```

### 1.3 核心设计原则

- **读写分离**：页面组件直接通过 Prisma 读取；数据写入统一走 API Route Handler。
- **前后台同构**：共享数据库连接、校验 schema、领域函数与类型定义。
- **多会议隔离**：绝大多数业务表均关联 `meetingId`，通过 `resolveMeeting()` 或当前会议 Cookie 确定数据作用域。
- **配置两级兜底**：`Meeting` 级站点配置优先，`SiteConfig`（固定 id=1）作为全局兜底。
- **权限模型简单**：仅区分 `USER` / `ADMIN` 两种角色，后台功能对所有管理员开放。

### 2.3 路由与会议切换机制

中间件 `src/proxy.ts` 负责路由重写与鉴权：

- `/` → 默认会议首页。
- `/m/{meetingId}` → 设置 `public_meeting_id` Cookie，进入该会议首页。
- `/m/{meetingId}/{path}` → 重写为 `/{path}?m={meetingId}`。
- `/r/{meetingId}` → 报名入口；未登录时携带 `callbackUrl` 跳转 `/login`。
- `/admin/:path*` → 受 `ADMIN` 角色保护，非管理员跳转登录页。

后台通过 `admin_meeting_id` Cookie 确定当前管理的会议；未选择会议时，除"会议管理"外其余侧边栏菜单禁用。

### 3.4 数据模型总览

数据库为 PostgreSQL，Prisma 层共定义 **23 个模型/数据表**：

| 类别 | 模型/表 | 说明 |
|---|---|---|
| 用户与认证 | `User`, `VerificationCode` | 账号信息、邮箱注册验证码 |
| 会议与配置 | `SiteConfig`, `Meeting` | 全局兜底配置、多会议主表 |
| 内容 | `Page`, `Notice` | 自定义页面、通知公告 |
| 报名与签到 | `RegistrationType`, `Registration`, `CheckinLog` | 报名类型、报名记录、签到日志 |
| 投稿 | `Submission` | 论文投稿 |
| 日程讲者 | `Speaker`, `Session`, `SessionSpeaker` | 讲者、场次、多对多关联 |
| 酒店 | `Hotel`, `HotelBooking` | 酒店、预订 |
| 图片直播 | `Album`, `Photo` | 相册、照片 |
| 嘉宾接待 | `Guest`, `GuestReception` | 嘉宾、接待安排 |
| 其他 | `BadgeTemplate`, `Channel`, `ChannelVisit`, `MeetingStaff` | 胸卡模板、渠道、访问记录、工作人员 |

核心关系：

- `Meeting` 是核心聚合根，与 `Page`、`Notice`、`Registration`、`Submission`、`HotelBooking`、`Guest`、`Channel`、`Speaker`、`Session`、`Hotel`、`Album`、`MeetingStaff` 为 1:N 关系；与 `BadgeTemplate` 为 1:1 关系。
- `User` 与 `Registration`、`Submission`、`HotelBooking` 为 1:N 关系；与 `Meeting` 通过 `MeetingStaff` 为多对多关系。
- `Registration` 与 `CheckinLog` 为 1:N 关系；与 `Meeting`、`User`、`RegistrationType`、`Channel` 为 N:1 关系。
- `Speaker` 与 `Session` 通过 `SessionSpeaker` 多对多关联，复合主键为 `[sessionId, speakerId, role]`，允许同一人以 SPEAKER/MODERATOR 两种角色参与同一场次。
- `Guest` 与 `GuestReception` 为 1:1 关系。

## 4. 业务场景核心操作流程

### 4.1 用户注册与登录

```
用户访问 /register
    → 填写姓名、邮箱、密码、验证码、联系方式（选填）、单位（选填）
    → 点击"获取验证码" → POST /api/auth/send-code
        → 校验邮箱唯一性，生成 6 位随机码写入 VerificationCode（3 分钟有效）
        → 通过 126 SMTP 发送邮件
    → 提交注册 → POST /api/register
        → zod 校验 → 校验验证码 → bcrypt 哈希密码 → 创建 User
    → 跳转 /login
        → 邮箱+密码登录 → NextAuth Credentials authorize
        → JWT 会话携带 id + role → 跳转 callbackUrl 或首页
```

### 4.2 会议报名流程

```
用户通过首页入口或 /r/{meetingId} 进入报名页
    → /r/{meetingId} 中间件强制登录（未登录跳转 /login?callbackUrl=...）
    → 报名页判断当前用户是否已报名该会议
        已报名：显示状态、参会类型、个人中心入口
        未报名：展示 RegistrationForm
    → 选择参会类型、填写姓名/单位/职称/电话
    → POST /api/registrations
        → 读取 channel_code Cookie 做渠道归因
        → 创建 Registration，状态 PENDING，生成唯一签到 token
    → 成功跳转 /m/{meetingId}/me
```

业务规则：

- 同一用户在同一会议下只能有一条报名记录（DB 级 `@@unique([userId, meetingId])`）。
- 报名状态：PENDING → APPROVED/REJECTED（管理员审核）。
- 报名成功生成唯一 `token`（uuid 去除连字符），作为签到凭证与胸卡二维码内容。

### 4.3 论文提交流程

```
登录用户进入 /submissions?m={meetingId}
    → 展示"我的投稿"列表
    → 填写论文题目、作者、摘要，上传 PDF（≤10MB）
    → POST /api/submissions
        → validatePdf() 校验文件类型与大小
        → savePdf() 保存到 public/uploads/{uuid}.pdf
        → 创建 Submission，状态 PENDING
    → 刷新页面
```

### 4.4 酒店预订流程

```
用户进入 /hotels?m={meetingId}
    → 展示该会议下所有酒店（价格/距离/地址/图片）
    → 选择酒店、入住/离店日期、房间数
    → POST /api/bookings
        → bookingSchema 校验（checkOut > checkIn，rooms 1-10）
        → 创建 HotelBooking，状态 PENDING
    → 成功跳转 /m/{meetingId}/me
```

### 4.5 管理员会议管理流程

```
管理员登录后台 → 未选择会议时强制跳 /admin/meetings
    → /admin/meetings 新建/编辑/删除会议、设为默认、选择管理
    → 选择会议后写入 admin_meeting_id Cookie
    → /admin 仪表盘可编辑会议级站点配置（名称、日期、地点、Logo、直播、欢迎页、脚页）
    → 其余菜单（通知、日程、讲者、报名等）均基于当前会议作用域操作
```

首个创建的会议会自动设为默认会议（`isDefault=true`）。删除会议会级联删除其下所有关联数据。

### 4.6 报名/投稿/预订审核流程

```
用户前台提交报名/投稿/预订 → 状态 PENDING
    → 管理员进入对应后台列表页
    → 查看详情，点击"通过"或"拒绝"
    → POST /api/admin/{registrations|submissions|bookings}/[id]
    → 更新 status 为 APPROVED 或 REJECTED
    → 可导出 CSV（toCsv 对 = + - @ 开头字段前置单引号，防公式注入）
```

### 4.7 日程与讲者管理流程

```
管理员先在 /admin/speakers 创建讲者（姓名、职称、单位、简介、照片、是否主持人）
    → 在 /admin/schedule 创建场次（日期、开始/结束时间、会场、标题）
    → 在场次编辑页关联讲者并指定角色 SPEAKER/MODERATOR
    → 前台 /schedule 按日期→会场→开始时间分组展示
```

讲者邀约与资料上传流程：

1. 管理员在 `/admin/speakers` 点击「邀约认证」，生成 `/s/{token}` 链接并分享给讲者。
2. 讲者访问 `/s/{token}`，登录后点击「接受邀约」，账号与讲者记录绑定（`Speaker.userId`）。
3. 已绑定讲者登录后，在 `/me` 个人中心看到「上传报告资料」入口。
4. 进入 `/me/speaker-materials`，选择关联日程并上传 PDF/PPT/PPTX，文件通过阿里云 OSS 保存，生成 `SpeakerMaterial` 记录。

### 4.8 签到流程

系统支持三种签到方式：

1. **自助签到**：个人中心生成二维码 `/checkin?token={registration.token}`，用户自行扫码 → `POST /api/checkin` → 方法 SELF。
2. **扫码签到**：管理员打开 `/admin/checkin/scan`，扫描参会者二维码 → `POST /api/admin/checkin` → 方法 SCAN。
3. **手动签到**：管理员在 `/admin/checkin/manual` 搜索姓名/手机/邮箱/单位 → `POST /api/admin/checkin` → 方法 MANUAL。

签到规则：

- 每次签到写入 `CheckinLog`。
- 首次签到更新 `Registration.checkedIn=true` 并记录 `checkedInAt`。
- 后续签到仍记录日志，但不再更新首次签到时间。

### 4.9 嘉宾邀请函确认流程

```
管理员在 /admin/guests 创建嘉宾 → 生成唯一 token
    → 复制邀请链接 /g/{token} 发送给嘉宾
    → 嘉宾打开链接查看会议信息、嘉宾信息、接待安排
    → 点击"确认出席" → POST /api/guests/{token}/confirm
    → 更新 Guest.confirmed=true，记录 confirmedAt
    → 管理员在 /admin/receptions 维护接待信息
```

### 4.10 渠道推广流程

```
管理员在 /admin/channels 创建渠道（短码、名称、负责人）
    → 分发专属链接 /register-conf?ch=CODE 或 /r/{meetingId}?ch=CODE
    → 访问时记录 ChannelVisit（IP、UA、sessionId）
    → 用户报名时读取 channel_code Cookie 写入 Registration.channelId
    → 后台 /admin/channels 统计 PV、UV、报名数、签到数、转化率
    → 可导出 CSV
```

### 4.11 胸卡导出流程

```
管理员进入 /admin/badge-template
    → 配置页面尺寸、背景图、姓名/职务/单位/二维码/会议标题坐标与字号
    → 预览 → POST /api/admin/badges/preview
    → 导出所有非拒绝状态报名者胸卡 → GET /api/admin/badges/export
        → 使用 pdf-lib 生成 PDF，qrcode 生成二维码（指向 /c/{token}）
        → 加载 public/fonts/simhei.ttf 支持中文
```

### 4.12 图片直播流程

```
管理员进入 /admin/albums
    → 创建相册（标题、日期、封面）
    → 上传照片到相册（图片 ≤5MB，jpeg/png/webp）
    → 前台 /photos 展示相册列表
    → 点击相册进入 /photos/[id] 查看照片
```

## 5. 系统现状与技术实现总结

### 5.1 当前已实现能力总结

本系统已覆盖一场学术/会务会议从筹备到结束的主要数字化环节：

- **多会议管理**：支持多个会议在同一实例中共存，通过默认会议与 `admin_meeting_id` Cookie 切换当前管理会议。
- **用户体系**：邮箱验证码注册、Credentials 登录、JWT 会话、`USER`/`ADMIN` 角色、`isActive` 账号启用/停用。
- **报名全流程**：在线报名、参会类型与费用、待审核/通过/拒绝状态、签到 token、渠道归因、CSV 导出。
- **论文投稿**：带 PDF 附件的投稿、单会议可投多篇、审核、导出。
- **酒店资源**：酒店信息发布、预订申请、审核、导出。
- **日程讲者**：讲者库、场次管理、SPEAKER/MODERATOR 双角色关联、前台按日期会场分组展示。
- **内容发布**：通知公告、自定义页面（venue/contact）、富文本页脚与欢迎致辞。
- **嘉宾接待**：嘉宾分级（VIP/NORMAL/MEDIA）、邀请函确认、接待安排（交通/酒店/车辆）。
- **签到体系**：自助/扫码/手动三种方式、签到日志、统计与导出。
- **图片直播**：相册与照片管理，作为会议现场资料沉淀入口。
- **胸卡打印**：模板化配置、批量 PDF 导出。
- **渠道统计**：短码推广、访问与转化统计。
- **用户管理**：跨会议报名记录查询、密码重置、角色控制。

### 5.2 当前数据持久化与归档现状

目前系统没有独立的"归档"动作或状态字段，数据留存与清理依赖于以下机制：

| 数据类型 | 存储位置 | 删除规则 |
|---|---|---|
| 会议元数据 | `Meeting` 表 | 删除会议级联删除其下所有数据 |
| 报名记录 | `Registration` + `CheckinLog` | 随会议级联删除；CheckinLog 随 Registration 级联删除 |
| 投稿记录与附件 | `Submission` + `public/uploads/` 文件 | 随会议级联删除；物理 PDF 文件不自动清理 |
| 酒店预订 | `HotelBooking` | 随会议级联删除 |
| 讲者/日程 | `Speaker` / `Session` / `SessionSpeaker` | 随会议级联删除 |
| 讲者资料 | `SpeakerMaterial` + 阿里云 OSS 文件 | 随讲者或日程级联删除；OSS 物理文件不自动清理 |
| 相册照片 | `Album` / `Photo` + `public/uploads/images/` 文件 | 随会议级联删除；物理图片文件不自动清理 |
| 嘉宾接待 | `Guest` / `GuestReception` | 随会议级联删除 |
| 渠道数据 | `Channel` / `ChannelVisit` | 随会议级联删除 |
| 胸卡模板 | `BadgeTemplate` | 随会议级联删除 |
| 自定义页面与通知 | `Page` / `Notice` | 随会议级联删除 |

关键约束：

- `User`、`RegistrationType`、`Hotel` 在被引用时不可删除（`onDelete: Restrict`）。
- `Meeting` 删除会级联删除几乎所有关联数据；`User`（owner）删除时，其拥有会议的 `ownerId` 置空。
- 物理文件（PDF、图片）与数据库记录未做联动清理，删除记录后文件可能残留。

### 5.3 当前技术边界与限制

以下为本系统当前实现的客观边界：

1. **会议生命周期无状态字段**
   - `Meeting` 表没有 `status` 或 `isArchived` 字段，无法标识"筹备中/进行中/已结束"。
   - 会议结束后需依赖管理员手动停止报名（如关闭报名入口或不再推广），系统不会自动禁用前台报名/投稿/预订。

2. **权限模型仅两角色**
   - 仅 `USER` 与 `ADMIN`，无会议级管理员、只读人员、财务人员等更细粒度角色。
   - 所有后台功能对 ADMIN 完全开放。

3. **物理文件管理未闭环**
   - 删除投稿 PDF、相册照片、胸卡背景图时，数据库记录删除，但 `public/uploads/` 下文件不会自动删除。
   - 未实现文件去重、缩略图、对象存储适配。

4. **无软删除机制**
   - 报名、投稿、预订、用户等关键业务表当前为硬删除，删除后不可恢复。

5. **历史数据跨会议查询入口有限**
   - 后台以"当前会议"为视角组织菜单；除 `/admin/users/[id]/registrations` 可查看单用户跨会议报名记录外，没有独立的跨会议聚合报表。

6. **报名表名默认待审核**
   - 无论 `Meeting.requireApproval` 如何设置，报名创建后状态均为 `PENDING`，需管理员手动审核为 `APPROVED`。

7. **MeetingStaff 已定义但未启用**
   - 数据模型中存在 `MeetingStaff` 多对多关系，但当前业务代码未使用，未实现会议工作人员授权。

### 5.5 给接手者的速查清单

- 想新增一个会议：访问 `/admin/meetings/new`，创建后"设为默认"或"进入管理"。
- 想修改前台内容：在 `/admin` 仪表盘改站点配置，或在 `/admin/pages`、`/admin/notices` 改页面/通知。
- 想开放报名：确保会议已配置至少一个 `RegistrationType`，用户即可通过 `/r/{meetingId}` 报名。
- 想生成签到二维码：报名审核通过后，用户个人中心 `/me` 会展示。
- 想导出数据：报名、投稿、预订、签到、渠道、接待、胸卡均有导出入口。
- 想确认数据隔离：绝大多数表都带 `meetingId`，切换当前会议 Cookie 即可查看不同会议数据。
- 新增后台接口时：首条语句必须调用 `isAdmin` 进行权限守卫，否则 `/api/admin/*` 存在越权风险。
- 修改数据模型后：更新 `prisma/schema.prisma`，运行 `npm run db:push`（开发环境），并视情况更新 `scripts/schema.sql`。
