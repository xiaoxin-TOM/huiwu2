# 响应式移动适配 设计文档

- 日期: 2026-07-01
- 状态: 已确认设计,待编写实现计划
- 背景: 现有会务系统是 HTML5 + Tailwind 网页,已部分响应式,但未为手机专门打磨。目标是在**手机浏览器**上也能顺畅使用,同时不破坏桌面端体验。

## 1. 目标与范围

给现有前后台补**响应式移动适配**。采用 **mobile-first + Tailwind 断点**:默认(base)样式面向手机,`md:`(≥768px)及以上恢复/保留桌面布局。

**不在本次范围(YAGNI):**
- 不做原生/混合 App(iOS/Android、React Native、Capacitor 等)
- 不做全套"卡片式表格"重设计(后台宽表格先用横向滚动保底)
- 不改任何业务逻辑、数据模型、API、鉴权
- 不做多会议/多租户(另行规划)

## 2. 技术约束

- Next.js 16(App Router,`src/`)。汉堡菜单等需要交互状态的部分抽成 `"use client"` 组件;页面主体尽量保持 Server Component。
- 仅用 Tailwind 既有断点(`sm` 640 / `md` 768 / `lg` 1024),不引入新依赖。
- 保持现有桌面端外观基本不变(`md:` 以上等价于现状)。

## 3. 改动点

### 3.1 前台顶部导航(`src/components/SiteHeader.tsx` + 新增客户端组件)
现状:Logo + 十余个导航链接横向 `flex-wrap` 平铺,右侧登录/个人中心。手机上挤成多行。

目标:
- **手机(<md)**:顶部仅显示 Logo + 右侧**汉堡按钮**;点击展开为 header 正下方的**竖向下拉面板**(非侧滑抽屉),含全部导航项 + 登录/个人中心入口;再次点击按钮或点击某项后收起。
- **桌面(≥md)**:保持现状(横排导航),隐藏汉堡按钮。
- 交互状态用一个新的 `"use client"` 组件(如 `MobileNav`)承载;导航项数据可由 `SiteHeader` 传入,登录态(session)仍在服务端读取后传入,避免客户端再取。
- 汉堡按钮、菜单项点击区 ≥44px。

### 3.2 后台侧边栏(`src/app/(admin)/admin/layout.tsx` + 新增客户端组件)
现状:固定宽 `w-52` 深色侧栏 + 内容区,横向排列。手机上侧栏占据过多宽度。

目标:
- **手机(<md)**:侧栏默认收起;后台顶部显示汉堡按钮,点击将侧栏以**抽屉**形式覆盖滑出(带半透明遮罩),点遮罩/菜单项收起;内容区全宽。
- **桌面(≥md)**:保持现状(固定侧栏常驻)。
- 抽屉开关用新的 `"use client"` 组件承载;`MENU` 数据不变。

### 3.3 后台宽表格(各 `admin/*/page.tsx` 列表页)
现状:报名、投稿、预订、用户、通知、讲者、日程、酒店等列表为 `<table>`,列多,手机上溢出。

目标:统一给每个宽表格套一层**横向滚动容器**:外层 `div` 加 `overflow-x-auto`(必要时表格设 `min-w-*`),使窄屏下表格可横向滑动而不撑破布局。仅结构性包裹,不改表格内容与逻辑。
(卡片式呈现留作后续可选优化,不在本次。)

### 3.4 全站断点梳理
- **栅格**:核对首页 Hero、酒店/讲者/相册/照片等 `grid`,窄屏收敛为单列或双列(多数已有 `sm:` 类,查漏补齐)。
- **间距与字号**:窄屏适当收敛(如大标题在手机上略小)。
- **点击区**:导航、按钮、表单控件在手机上点击区 ≥44px(必要处加 `py`/`min-h`)。
- **表单**:手机端单列(现状多为 `max-w-*` 单列,核对报名/投稿/预订/后台各表单)。
- **图片**:本地 `<img>` 确保 `max-w-full`,不横向溢出。

### 3.5 viewport / 元信息(`src/app/layout.tsx`)
- 确认/显式导出 `viewport`(`width=device-width, initial-scale=1`)。Next 16 默认会注入,但显式声明更明确。不改其它 metadata。

## 4. 组件与边界

- `MobileNav`(新,client):前台移动导航开关 + 菜单面板。输入:导航项数组、是否已登录(布尔)。职责单一:仅负责移动端菜单的展开/收起与渲染。
- `AdminSidebar`/`AdminShell`(新,client):后台移动抽屉开关 + 侧栏渲染。输入:菜单项数组。职责单一:移动端抽屉行为。
- 现有页面主体保持 Server Component;只有上述两个交互壳是 client。

## 5. 测试策略

响应式以**视觉验收**为主:
- 用 Chrome DevTools 在 **375 / 768 / 1024** 三档逐页核对:首页、通知列表/详情、简明/详细日程、讲者列表/详情、会场交通、联系方式、报名、投稿、个人中心、酒店、图片直播、直播,以及后台各管理页。
- 关键交互(前台汉堡菜单、后台抽屉)在浏览器/真机点开点关验证;`npm run build` 保持通过。
- 自动化:仅对新抽出的 `MobileNav` / 后台抽屉组件做**开合状态**的小型组件测试(如初始收起、点击切换)。其余不强求单测。

## 6. 实现分期(粗估 1.5–2 天)

1. 前台导航汉堡菜单(`MobileNav` + `SiteHeader` 改造)
2. 后台侧栏抽屉(`AdminShell` + admin layout 改造)+ 后台宽表格横向滚动
3. 全站断点梳理(栅格/间距/字号/点击区/图片/表单)+ viewport 核对
4. 375/768/1024 三档逐页视觉验收 + 收尾

每期可独立验证。
